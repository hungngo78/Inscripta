System Requirements and User Stories

High level (epic) requirement

Build out this system, so that a user can create new edit insertion requests and view previously created edit requests.

System Requirements

  • Build a front end using the starter Angular project
    • Feel free to use Angular Material components or any other open source UI component package you might prefer.
    • CSS Styling is not the main focus of this exercise
  • Build out back end web service from the provided Spring Boot (Java 8) starter project
    • Build out RESTful endpoints and appropriate class structures to support front end data and persistence
    • NOTE - There is a java package called ucsc that already knows how to query a public API to fetch data for our target genome sacCer3.
      • It can fetch chromosomes for this genome and gene definitions with their start/stop positions within a given chromosome.
      • This package does not need to be altered. It can be considered readonly.
  • Any unit tests must pass

User Stories

  • As a User, I can create a Gene Insertion Edit request

    Acceptance Criteria
    • User can navigate to a screen using the menu where new Edit Requests can be created
    • User is presented with a dropdown of possible chromosomes
    • When the user selects a chromosome, they are presented with valid genes within that chromosome
    • When the user selects a gene, they have the ability to enter a relative position within the gene.
    • This relative position must be validated, based on start/stop positions of the selected gene provided by the UCSC Gene API
      • (e.g. if a gene hypothetically has a start position of 100 and an end of position of 150, then there are 50 possible valid positions to insert at (between position 1 and 50 of the selected gene)
    • User has the ability to enter one or more Nucleotides (must be one of the following: “A”,”C”,”T”, or “G”) to insert into the gene
    • After the inputs have been validated, the user can save their edit request (API call to backend)
      • In lieu of a database, the data can be saved to flat file on the server side local disk (or to save time, you could persist this to application scoped memory)
      • Generate a unique identifier for the edit request
      • In addition to saving the inputs from the user, the “target” gene record must be saved alongside.

  • As a User, I can view previous Gene Insertion Edit requests

    Acceptance Criteria
    • User can view previous Insertion Edit Requests in a table/grid format
    • Each record must have:
      • Unique identifier
      • Gene name
      • Gene start
      • Gene stop
      • Edit position
      • Nucleotide(s) Inserted
      • Datetime saved
    • After edit requests are saved, this table should refresh with latest values
